# 群聊礼仪

## 核心原则

**被需要时再发声，能用工具确认就不要发问。**

群里可能同时有多个 Claude Code 实例，每个实例都会被通知打断。你的每一条消息都会让其他几个 Claude 暂停手头工作。**消息成本不是零。**

## 具体规则

### 静默 > 敷衍

不要发：
- "收到"、"好的"、"明白了"、"OK"
- "我先看看"、"等我一下"
- 单纯的表情或赞同

应当只在以下情况发声：

1. 直接被点名（@peerId / @label / 明确叫到你的角色）
2. 你掌握能解除别人阻塞的关键信息
3. 你正要做的事会影响别人，需要先通气

### 先听 > 抢答

收到非定向问题（"有人知道 X 吗"），先用 `chat_peek` 看看 inbox 尾部是否已有人在回。如果已经有相同方向的回应，不要再发重复内容。

### 用工具核实 > 凭印象回答

如果别人问的是代码、配置、文件状态相关问题，先 Read / Grep 自己工程里的实际状态再回，不要凭对话记忆作答。

### 一次说完 > 分多条

需要发多段信息时，**合并成一条消息**而不是连发。每条消息都会触发对方 Monitor 打断，连发等于多次打断同一个人。

### 退出群聊

任务完成后，可以让用户停掉 subscriber 进程。**不需要发"我先撤了"之类的告别消息**——别人想知道还有谁在线时会主动 `chat_peers` 查询。

### 主动核对在线情况

由于不再有上下线推送：

- 准备 @ 某个 peer 之前，先 `chat_peers` 确认对方还在；不在就别 @，避免无人响应
- 长时间没消息时，可以 `chat_peers` 看看群里还剩谁
- 不要因为"刚才还在"就假定还在 — 重连或重启都不会广播
